Yeah, I’m going to be blunt: dc3dd is failing because you’re using the wrong syntax (you keep typing bs= and count= like GNU dd). dc3dd does NOT use bs= or count=. It uses:

  • ssz= = sector size (bytes)
  • cnt= = number of sectors to read (not “blocks”)
  • bufsz= = internal buffer size (optional)
  • plus if=, of=, hash=, log= ✅

So stop fighting it. Copy/paste this exactly.

Get dc3dd working (copy/paste, no improvising)

0) First, make sure your E01 mount is present

sudo mkdir -p /mnt/ewf
sudo umount /mnt/ewf 2>/dev/null || true
sudo ewfmount ~/cases/usb_a.E01 /mnt/ewf
sudo ls -la /mnt/ewf

You must see a file called ewf1 in /mnt/ewf (that’s the “raw view” of the E01). If you don’t see ewf1, dc3dd cannot work.

1) Make a small forensic image sample (fast) + hash + log

This copies 64 MiB from the start of the evidence (safe + quick). Why 64 MiB? Because cnt=131072 sectors × ssz=512 bytes = 67,108,864 bytes (~64 MiB).

mkdir -p ~/dc3dd_out
sudo rm -f ~/dc3dd_out/sample.dd ~/dc3dd_out/dc3dd_sample.log ~/dc3dd_out/sample.dd.sha256

sudo dc3dd if=/mnt/ewf/ewf1 \
  of=/home/vboxuser/dc3dd_out/sample.dd \
  hash=sha256 \
  log=/home/vboxuser/dc3dd_out/dc3dd_sample.log \
  ssz=512 \
  cnt=131072

2) Verify outputs (this is your “proof”)

ls -lh ~/dc3dd_out/sample.dd
tail -n 25 ~/dc3dd_out/dc3dd_sample.log
sha256sum ~/dc3dd_out/sample.dd | tee ~/dc3dd_out/sample.dd.sha256

What you did wrong (so you stop repeating it)

  • count= → dc3dd wants cnt= (sectors).
  • bs= → dc3dd doesn’t use it; use bufsz= (optional) and ssz=.
  • You also typed /mnt.ewf sometimes — that’s a different path. It must be /mnt/ewf.

Your own screenshots literally show “unrecognized option bs=…” and “unrecognized option count=…”. That’s not “Linux being annoying”, that’s you using the wrong tool syntax.
